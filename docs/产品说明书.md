## 🎯 产品概述

### 1.1 产品简介

DataSec Hub 是一个专业的**数据安全合规知识服务平台**，通过创新的 **COU（合规义务单元）** 体系，将复杂的法律法规转化为清晰、可执行的合规要求，帮助企业实现数据合规的智能化管理。

### 1.2 核心价值主张

> **"让数据合规更简单"**

- 📚 **全面**：覆盖国内所有数据安全相关法律法规、标准规范
- 🎯 **精准**：COU体系将模糊条款转化为明确义务
- ⚡ **高效**：智能场景构建，快速生成定制化合规方案
- 🔍 **可查**：全文检索、多维度筛选、版本追溯
- 🔌 **可集成**：标准化API，支持企业系统集成

### 1.3 产品愿景

成为中国企业数据合规的基础设施，让每一家企业都能轻松应对数据安全合规挑战。

## 💡 核心创新体系

### 3.1 COU（Compliance Obligation Unit）体系

**核心理念**：将复杂的法律条款转化为结构化的合规义务单元

```
COU示例：COU-DSL-001
├── 编码：COU-DSL-001
├── 标题：重要数据处理者应当明确数据安全负责人和管理机构
├── 来源：《中华人民共和国数据安全法》第27条
├── 类型：强制性
├── 基础权重：10分（法律级别）
├── 适用范围：
│   ├── 行业：通用
│   ├── 区域：国内
│   └── 用户类型：大型企业、关基运营者
├── 要求行动：
│   ├── 1. 确定数据安全负责人
│   ├── 2. 建立数据安全管理机构
│   └── 3. 落实数据安全保护责任
├── 技术措施：[组织架构设计、责任制度]
├── 组织措施：[管理制度、培训计划]
└── 关联COU：[COU-DSL-002, COU-PIPL-015]
```

### 3.2 四层架构解耦设计

```
┌──────────────────────────────────────────────────────────┐
│  Layer 4: COU（合规义务单元）- 场景化实例                  │
│  - 针对具体业务场景的定制化合规要求                        │
│  - 继承自ControlObjective，支持场景化调整                  │
└──────────────────────────────────────────────────────────┘
                              ↑ 引用/定制
┌──────────────────────────────────────────────────────────┐
│  Layer 3: ControlObjective（控制目标）- 标准控制库        │
│  - 200-500个可维护的标准控制目标                           │
│  - 多对一映射：多个法规条款 → 一个控制目标                 │
│  - 提供场景无关的标准控制要求                              │
└──────────────────────────────────────────────────────────┘
                              ↑ 映射
┌──────────────────────────────────────────────────────────┐
│  Layer 2: Clause（法规条款）- 原文结构化                   │
│  - 法律法规的具体条款                                      │
│  - 映射到ControlObjective                                  │
│  - 版本追踪和变更管理                                      │
└──────────────────────────────────────────────────────────┘
                              ↑ 包含
┌──────────────────────────────────────────────────────────┐
│  Layer 1: Policy（政策文件）- 法规元数据                   │
│  - 法律法规、标准规范的元信息                              │
│  - 版本管理、状态追踪                                      │
│  - 关联关系管理                                            │
└──────────────────────────────────────────────────────────┘
```

### 3.3 五维标签系统

| 维度 | 代码前缀 | 描述 | 示例 |
|------|----------|------|------|
| **客体维度** | OBJ-xxx | 保护对象 | 个人信息、重要数据、国家核心数据 |
| **主体维度** | SUB-xxx | 责任主体 | 处理者、运营者、监管者 |
| **业务流转维度** | LIF-xxx | 业务环节 | 收集、存储、使用、加工、传输、删除 |
| **安全域维度** | SEC-xxx | 保护手段 | 加密、访问控制、审计、治理 |
| **动作义务维度** | ACT-xxx | COU核心 | 文档化、评估、培训、审计、报告 |

### 3.4 三层权重计算体系

**公式**：`最终权重 = 基础权重 × 罚则权重 × 标签匹配度 × 场景权重`

| 权重类型 | 计算依据 | 分值范围 |
|----------|----------|----------|
| **基础权重** | 政策级别（法律10分 → 指南指引4分） | 4-10 |
| **罚则权重** | 违规后果（刑事5分 → 无1分） | 1-5 |
| **标签匹配度** | 场景标签与COU标签的匹配程度 | 0-1 |
| **场景权重** | 行业/区域/用户类型匹配系数 | 0.6-1.0 |

**优先级分级**：
- 🔴 **关键** (权重 9-10)：必须立即实施
- 🟠 **高** (权重 7-8)：强烈建议实施
- 🟡 **中** (权重 5-6)：根据实际情况实施
- 🔵 **低** (权重 1-4)：可选实施
